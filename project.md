# ToDoT 프로젝트 설계 문서

## 1. 프로젝트 개요

| 항목 | 내용 |
|------|------|
| 프로젝트명 | ToDoT |
| 목적 | 이미지/동영상을 다양한 아트 스타일(픽셀, 문자, 아트 효과)로 변환하는 데스크톱 프로그램 |
| 플랫폼 | Windows 7 이상 |
| 인터페이스 | GUI (PyQt5) + CLI |
| 산출물 | 실행 파일(.exe), 사용 가이드, 빌드 스크립트 |

## 3. 사용자 시나리오

### S1: 이미지 픽셀아트 변환
1. 사용자가 이미지를 열기(Ctrl+O) 또는 드래그 앤 드롭으로 로드
2. 카테고리 "픽셀 변환" 선택 → 세부 모드 선택 (예: 디더링)
3. 픽셀 크기/색상 수 슬라이더 조절 → 실시간 미리보기 자동 갱신
4. "변환하기" 클릭 → 원본 해상도 변환 실행
5. "저장" 클릭 → output 폴더에 저장, 히스토리 갱신

### S2: 동영상 변환
1. 동영상 파일 로드 → 원본 재생 패널 표시
2. 모드/파라미터 변경 → 5초 미리보기 자동 생성
3. "변환하기" 클릭 → 프레임별 변환, 진행률 + 시간 표시
4. 완료 시 결과 동영상 자동 재생, output 폴더에 저장

### S3: 복합 모드
1. 카테고리 "복합 모드" 선택
2. 우상단/좌하단/우하단 3개 패널에 각각 변환 모드 지정
3. 변환 → 원본(좌상단) + 3개 변환 결과가 2x2로 합성된 이미지 생성

### S4: 아트 효과 적용
1. 카테고리 "아트 효과" 선택 → 세부 모드 (보로노이, 만화 등) 선택
2. 크기 슬라이더로 디테일 정도 조절 → 실시간 미리보기
3. 변환 및 저장

### S5: CLI 일괄 변환
```
python src/cli.py input.png --mode voronoi --pixel-size 6 -o result.png
python src/cli.py video.mp4 --mode cartoon --pixel-size 10
python src/cli.py photo.jpg --mode composite --composite-modes pixel halftone lowpoly
```

## 5. UI 설계

### 5.1 전체 레이아웃

```
┌──────────┬──────────────────────────────────────────────┐
│ 좌측 패널 │  [이미지 미리보기 | 새 그림 | 동영상 미리보기] │
│ (320px)  │                                              │
│          │  ┌──────────────┬───────────────┐            │
│ [결과물]  │  │   원본       │   변환 결과   │            │
│ [히스토리]│  │              │               │            │
│          │  └──────────────┴───────────────┘            │
│          ├──────────────────────────────────────────────┤
│          │  Row1: [카테고리▾] [세부모드▾] [팔레트▾]       │
│          │  Row2: [크기 ═●] [색상] [격자] [윤곽]  [변환] │
└──────────┴──────────────────────────────────────────────┘
```

### 5.2 변환 패널 (2행 구조)

**Row 1 - 모드 선택:**
- 카테고리 콤보: 픽셀 변환 | ASCII 변환 | 한글/유니코드 | 아트 효과 | 복합 모드
- 세부 모드 콤보: 카테고리에 따라 동적 변경
- 팔레트 콤보: pixel_palette 모드에서만 표시
- 복합 모드: 우상단/좌하단/우하단 3개 모드 선택 콤보

**Row 2 - 파라미터 + 액션:**
- 크기 슬라이더 (2~32): 공통 파라미터 (모든 모드에서 블록/디테일 크기 역할)
- 색상 수 (2~256): K-means 계열 모드 전용
- 격자/윤곽선: 픽셀 모드 전용
- 취소/변환/저장 버튼

### 5.3 파라미터 가시성 규칙

| 카테고리 | 크기 | 색상 | 격자 | 윤곽선 | 팔레트 |
|----------|------|------|------|--------|--------|
| 픽셀 변환 | ✅ | 모드별 | ✅ | K-means만 | palette만 |
| ASCII 변환 | ✅ | ❌ | ❌ | ❌ | ❌ |
| 한글/유니코드 | ✅ | ❌ | ❌ | ❌ | ❌ |
| 아트 효과 | ✅ | ❌ | ❌ | ❌ | ❌ |
| 복합 모드 | ✅ | 모드별 | ✅ | ✅ | ❌ |

## 6. 변환 모드 상세

### 6.1 픽셀 변환 (6종)

| 모드 | 알고리즘 | 주요 파라미터 |
|------|----------|---------------|
| K-means 양자화 | 다운스케일 → K-means 색상 클러스터링 → 업스케일 | pixel_size, num_colors, grid, outline |
| 디더링 (Floyd-Steinberg) | K-means 팔레트 추출 후 Floyd-Steinberg 오류 확산 | pixel_size, num_colors |
| 팔레트 매핑 | 사전 정의 팔레트에 최근접 색상 매핑 (벡터화) | pixel_size, palette_name |
| Nearest Neighbor | 순수 INTER_NEAREST 다운/업스케일 | pixel_size |
| Edge-preserving | bilateral filter → Canny 엣지 → K-means → 엣지 오버레이 | pixel_size, num_colors |
| 슈퍼픽셀 | pyrMeanShiftFiltering → K-means 양자화 | pixel_size, num_colors |

### 6.2 문자 아트 (4종)

| 모드 | 문자 세트 | 셀 크기 |
|------|-----------|---------|
| ASCII (컬러/흑백) | 70자 밝기 매핑 | 10x18 |
| 한글 (컬러/흑백) | 자모+음절 32자 | 14x20 |
| 유니코드 (컬러/흑백) | 블록 문자 15자 | 12x18 |

### 6.3 아트 효과 (6종)

| 모드 | 알고리즘 | pixel_size 역할 |
|------|----------|-----------------|
| 하프톤 | 블록별 밝기 기반 원형 도트 크기 조절, 원본 컬러 | dot_size |
| 만화 스타일 | bilateral filter N회 + adaptiveThreshold 엣지 합성 | 스무딩 횟수 + 엣지 블록 |
| 보로노이 | Canny 엣지 + 랜덤 시드 → Subdiv2D 보로노이 → 셀 평균색 | 셀 개수 (역비례) |
| 로우 폴리 | Canny + 랜덤 → Delaunay 삼각분할 → 삼각형 평균색 | 삼각형 수 (역비례) |
| 점묘화 | 밝기 역비례 확률 기반 랜덤 도트 배치, 원본 컬러 | 도트 밀도 + 크기 |
| 글리치 | 채널 시프트 + 블록 변위 + 스캔라인 + 색상 파괴 | 강도 |

### 6.4 복합 모드

- 입력 이미지를 4분할: 좌상단(원본), 우상단/좌하단/우하단(각각 선택한 모드로 변환)
- 각 패널은 원본과 동일 크기로 리사이즈 후 합성
- 결과 이미지 크기: 원본의 2배 x 2배

## 7. 제약사항 및 미구현 사항

### 구현하지 않은 모드

| 모드 | 사유 |
|------|------|
| 벡터 → 픽셀 | SVG/AI 벡터 파일은 래스터와 다른 포맷. cairosvg 등 별도 라이브러리 + 입력 파이프라인 전면 변경 필요 |
| Photomosaic | 알고리즘 자체는 단순하나, 수백~수천 장의 타일 이미지 라이브러리가 필수. 현재 "1장 입력 → 1장 출력" 아키텍처와 부합하지 않으며, 타일 라이브러리 관리 UI/워크플로우 설계가 별도 프로젝트급 |

### 알려진 제한

- 보로노이/로우폴리/점묘화 모드의 동영상 변환: 프레임마다 랜덤 시드가 달라 약간의 플리커링 발생 가능
- 디더링 모드: 고해상도 이미지에서 Python 루프 사용으로 변환 속도 느림 (미리보기는 축소본 사용으로 무관)
- 만화 스타일: bilateral filter 다중 패스로 고해상도 동영상 변환 시 속도 저하

## 8. AI 활용 개발 가이드

### 8.1 프롬프트 설계 원칙

1. **기획 문서 우선**: `project.md`를 컨텍스트로 항상 제공하여 AI가 전체 설계를 이해한 상태에서 코딩
2. **점진적 확장**: 기본 기능(픽셀 변환) → 추가 모드 → UI 개선 → 고급 효과 순서로 요청
3. **변경 범위 명시**: "converter.py에 새 함수 추가", "main_window.py의 convert panel 수정" 등 파일 범위 지정
4. **테스트 포함 요청**: 구현 후 "테스트 실행해서 확인" 명시

### 8.2 반복 개발 전략

```
[기획 문서 작성] → [AI에게 기본 구조 생성 요청]
       ↓
[코드 리뷰 에이전트 실행] → [문제점 파악]
       ↓
[개선 사항 반영 요청] → [테스트 + 빌드 확인]
       ↓
[기능 추가 요청 (새 모드 등)]
       ↓
[문서 업데이트 요청 (README, project.md)]
```

### 8.3 코드 리뷰 방법론

- **듀얼 에이전트 리뷰**: 코드 품질 에이전트 + UX 리뷰 에이전트를 동시 실행하여 교차 검증
- **리뷰 관점**: 스레딩 안전성, 에러 핸들링, UI 응답성, 한글 호환성, 인코딩 이슈
- **반영 기준**: 영향도(High/Medium/Low) × 구현 난이도(Low/Medium/High) 매트릭스로 우선순위 결정

### 8.4 문서 동기화 규칙

- **수정 발생 시 문서 반드시 갱신**: 새 모드 추가 → converter.py + main_window.py + cli.py + project.md + README.md
- **배치 파일 인코딩**: `.bat` 파일에는 한글 사용 금지 (CP949/UTF-8 충돌 방지)
- **커밋 전 체크리스트**: lint 확인 → import 확인 → 빌드 테스트 → 문서 동기화
